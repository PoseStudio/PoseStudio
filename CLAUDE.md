# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

PoseStudio is an early-stage, open-source Qt6/C++ desktop application for 3D character creation, posing, and scene building (think a free alternative to DAZ Studio). The codebase is currently a small skeleton: a main window shell, an asset manager side panel backed by SQLite, a Preferences dialog, and a menu bar. The 3D viewport, rigging, animation, and rendering systems described in README.md are aspirational/roadmap — they don't exist in code yet.

## Build

Requires Qt 6 (developed against 6.11.1, MSVC 2022 64-bit) and CMake 3.17+.

```
cmake -B build -DCMAKE_PREFIX_PATH="C:/Qt/6.11.1/msvc2022_64"
cmake --build build
```

If `CMAKE_PREFIX_PATH` isn't passed, `CMakeLists.txt` falls back to the hardcoded path `C:/Qt/6.11.1/msvc2022_64` — never commit a change to that fallback to point at your personal Qt install.

The build links `Qt6::Widgets` and `Qt6::Sql` only (no Qt Network/3D/Quick modules yet). `CMAKE_AUTOMOC`/`AUTOUIC`/`AUTORCC` are all on, so new `Q_OBJECT` classes, `.ui` files, and resources just need to be added to the `add_executable` source list / `resources.qrc` — no manual moc/rcc invocation.

A post-build step mirrors `resources/Maquettes/` (the built-in asset library, real on-disk content — not embedded via `resources.qrc`, since the Asset Manager browses libraries through `QDir`, not Qt's resource filesystem) next to the executable on every build. It's a true mirror (`rm -rf` then `copy_directory`), not an overlay, so files renamed/removed in source don't linger in the deployed copy.

There is no test suite, linter, or CI configuration in this repo yet.

## Architecture

**Startup sequence matters** (`src/main.cpp`): the SQLite database is opened and `PreferencesManager` is loaded *before* any widget is constructed, because widgets (e.g. `AssetManagerWidget`) read preferences/DB state during construction. If you add a new manager/cache that UI depends on, initialize it in this same early block, not lazily inside a widget.

**Database** (`src/database.{h,cpp}`): single named QSqlDatabase connection (`"db_conn"`) shared process-wide via `QSqlDatabase::database("db_conn")` rather than opened per-call. The SQLite file lives next to the executable (`posestudio.db`), not in the working directory or a user-profile path. Schema lives in `resources/database/initialize.sql`, embedded into the binary via `resources.qrc`, and is executed both on `DbInitMode::FactoryReset` (which also deletes the existing file first, and is reachable from the app via Preferences → Factory Reset) *and* on a genuine first launch (no `posestudio.db` file yet) — `initializeDatabase()` checks `QFile::exists(dbPath)` *before* FactoryReset can delete it, so a fresh install always gets the schema even via `DbInitMode::Normal`. There is still no formal migration system, but additive schema changes for existing databases (new columns/tables) are applied unconditionally at the end of `initializeDatabase()`, each guarded to be a no-op once already present (e.g. a `PRAGMA table_info` check before `ALTER TABLE ADD COLUMN`, or `CREATE TABLE/INDEX IF NOT EXISTS`) — add new ones here rather than only in `initialize.sql`, or pre-existing installs will silently fail at the first query touching the new column/table.

`AssetLibraries` has an `AssetLibraryIsBuiltIn` flag for the one library that ships with the app ("Maquettes", at `<applicationDirPath>/Maquettes`): `initializeDatabase()` re-syncs that row's path to the current install location on every launch (in case the app moved) and `mkpath`s the folder as a safety net. Built-in libraries always sort first (`ORDER BY AssetLibraryIsBuiltIn DESC`) and are excluded from the user-manageable list in Preferences → Assets, but still appear/behave like any other library in the Asset Manager tree.

`Favorites` (`FavoriteID`, `FavoritePath` unique, `FavoriteSortOrder`) is a flat list of favorited asset file paths — no folders, no nesting. `FavoriteSortOrder` holds the user's manual drag order from the Favorites pane (see Asset Manager section below); new favorites append after the current max.

**Preferences manager** (`src/preferencesmanager.{h,cpp}`): a process-wide singleton (`PreferencesManager::instance()`) that loads the entire `Preferences` SQLite table into a `QHash` once at startup. Reads never touch the DB; writes update the cache and persist immediately. It is explicitly not thread-safe — only call it from the Qt GUI thread.

**Asset manager** (`src/assetmanagerwidget.{h,cpp}`, the largest file in the project): the side-panel widget that browses physical asset library folders (paths stored in `AssetLibraries`) alongside virtual, user-defined "Collections" (`AssetCollections`/`AssetCollectionItems`/`AssetCollectionFolders` tables) and a flat "Favorites" list (`Favorites` table). Collections can nest arbitrarily deep via `AssetCollections.AssetCollectionParentID` (`0` = top-level) — any collection, at any depth, can hold its own flat asset items and folder shortcuts alongside further sub-collections, mirroring filesystem folder semantics. Favorites, by contrast, is intentionally flat — no folders, no sub-favorites, just asset items. `contextForTreeItem()` walks a tree item's ancestor chain to classify it as `Library`/`Collection`/`SearchResults`/`Favorites` regardless of nesting depth; prefer it over checking a node's immediate parent when context-menu or grid logic needs to know which "world" an item belongs to. Tree root order is Search Results, Favorites, Collections, then physical libraries (built-in libraries first). Key pieces:
- `AssetFolderProxyModel` wraps the filesystem `QStandardItemModel`-backed tree to mark folders as direct/indirect "hits" (i.e. contain or descend into a folder containing matching assets) for search/filtering, with results debounced via a pending-index queue rather than recomputed synchronously.
- `AssetTreeDelegate` / `AssetGridDelegate` fully override Qt's default painting for the directory tree and the thumbnail grid respectively — don't expect standard `QStyledItemDelegate` behavior; geometry constants they rely on (icon size, margins, gaps) are centralized in `src/constants.h` so the two delegates' size hints can't drift apart.
- Breadcrumb rendering is cached per `displayFolder()` call (`m_breadcrumb*` members) and only re-laid-out, not recomputed, on label resize. Virtual sources (Collections, Favorites) instead show a plain title (`m_currentTitleText`); `displayFolder()` and `refreshTitleLabel()` both branch on the same "is this a virtual path" check (`COLLECTION_` prefix or `FAVORITES_ROOT`) — keep them in sync if a third virtual source is ever added.
- Physical folders and DB-backed virtual Collections/Favorites are all rendered into the same `QTreeView`/`dirModel`, distinguished by `searchResultsRootItem`/`favoritesRootItem`/`collectionsRootItem` root nodes — when adding tree behavior, check which subtree (filesystem vs. virtual) an index belongs to before assuming filesystem semantics (e.g. rename-on-disk) apply.
- `buildAssetHits()` resolves a flat list of asset paths into thumbnailed `AssetHit`s (shared by `parseCollectionAssets`/`parseFavorites`); it groups by folder internally for efficient thumbnail scanning but always re-emits in the caller's *input* order — both Favorites and Collections rely on this to preserve the user's manual drag order, so don't reintroduce a hash-ordered return here.
- Both Favorites and Collections are sortable grids backed by their own per-row order column (`Favorites.FavoriteSortOrder`, `AssetCollectionItems.AssetCollectionItemSortOrder`); `AssetManagerWidget::isSortableView()` (true when `m_currentFolderPath` is `FAVORITES_ROOT` or a `COLLECTION_*` id) is the single gate for both the drag-reorder handling and the "Sortable" info-bar label — check it rather than re-deriving the same `FAVORITES_ROOT`/`COLLECTION_` checks elsewhere. The drag-to-reorder itself (`beginGridDrag`/`endGridDrag`/`gridInsertIndex`/`updateGridDropIndicator`/`persistGridOrder` + the `MouseButtonPress/Move/Release` block in the grid viewport's `eventFilter`) is a **hand-rolled** drag, not Qt's `QListWidget` `InternalMove`: in `IconMode`, Qt's built-in internal-move free-positions the icon instead of reordering rows, and its `QDrag` runs through the OS drag loop. The manual version tracks raw mouse events, floats a translucent ghost `QLabel` plus a between-items drop-line indicator, leaves the grid untouched until release (moving items mid-drag would reflow the layout under the cursor), and auto-scrolls via a `QTimer` when the cursor sits in the top/bottom edge zone. `persistGridOrder()` branches on which view is current to write back to the right table/column. `AssetGridDelegate::paint` checks a `gridDragging` dynamic property on the grid to suppress the hover highlight while this is active.

**Styling**: dark theme enforced via `app.setStyle(new AppProxyStyle("Fusion"))` in `main.cpp`, never the native OS style. `AppProxyStyle` (`src/appproxystyle.h`) must be constructed with `"Fusion"` explicitly — a bare `AppProxyStyle()` wraps the platform *default* style (e.g. `QWindows11Style` on Windows) instead, silently defeating the whole point of forcing Fusion. All app-level theming (style + global tooltip animation toggles) lives in this one spot in `main.cpp`'s "Branding & theming" block, not scattered into widget constructors. Stylesheets are modular (`global.qss`, `_assetmanager.qss`, `_menumanager.qss`, `_preferences.qss`) and concatenated in `loadStylesheets()` (`src/main.cpp`) in a fixed cascade order — later files override earlier ones, so module-specific overrides belong in the `_*.qss` files, not `global.qss`. Some custom widgets (e.g. `AssetTreeView`) expose `Q_PROPERTY` values (like `separatorColor`) specifically so QSS can drive C++ painting logic. `QDialog`/`QMessageBox` are dark-themed globally too — but note Qt's QSS engine doesn't apply selectors (e.g. `a { color: ... }`) to rich-text `<a>` links inside a `QLabel`; those need the color set inline in the HTML string itself (see the "Add Asset Folder" hint label in `assetmanagerwidget.cpp` for the pattern). Also note: avoid declaring *any* `border` (even `border: none;`) on a `QListView`/`QListWidget` `::item` rule unless you actually want one — doing so flips Qt to full CSS box-model rendering of that item, which can visibly inflate the selection/hover rect beyond the item's actual layout cell (this is why `#AssetManagerTree::item` and `#AssetLibraryList::item` deliberately have no `border` line). `outline: none;` is fine on a container (`#AssetManagerTree`, `#AssetLibraryList`) to kill the dotted focus ring, but don't assume it's a no-op everywhere in the cascade.

**Destructive operations** (e.g. Preferences → Factory Reset, in `factoryresetpreferencespanel.cpp`): rather than trying to reconcile every widget's in-memory state (the asset tree, `PreferencesManager`'s cache, etc.) after wiping the database, the app just relaunches itself via `QProcess::startDetached(QApplication::applicationFilePath(), ...)` followed by `QApplication::quit()`. Follow this pattern for any future action that invalidates broad in-memory state rather than writing ad-hoc refresh logic for every affected widget. The reset itself is gated inline on the page (type "RESET" into a text field to enable the button) rather than behind a separate confirmation popup.

**Preferences dialog** (`src/preferencesdialog.{h,cpp}`, `src/preferencespanel.{h,cpp}`, `src/*preferencespanel.{h,cpp}`): a modal `QDialog` with a vertical tab list (left) driving a `QStackedWidget` (right). Each tab is a `PreferencesPanel` subclass living in its own file (`AssetsPreferencesPanel`, `FactoryResetPreferencesPanel`, etc.) so areas can be built out independently — adding a tab means adding a new subclass file and one `addPanel()` call in `PreferencesDialog`'s constructor, not touching the others. `PreferencesPanel` provides shared chrome (heading + top-aligned content layout) plus `addDescription()`/`addPlaceholder()` helpers for standardized body-copy styling — use these instead of hand-rolling a `QLabel` with inline font-size, or the panel won't match the others. `MenuManager::openPreferencesDialog(initialTab)` is the one entry point that opens the dialog (Edit → Preferences, and the Asset Manager's "Manage Asset Folders" context actions both route through it) and wires the `AssetsPreferencesPanel`'s `librariesChanged`/`navigateToLibraryRequested` signals back to the live `AssetManagerWidget` — `MenuManager::setAssetManagerWidget()` must be called (from `main.cpp`, after the Asset Manager tab exists) before that wiring can attach, so don't move `setupMenus()` to run after it without checking this dependency.

**Constants** (`src/constants.h`): shared UI dimensions, colors, and timing values live here rather than being duplicated as magic numbers across widgets/delegates — add new shared layout/color/timing constants here instead of inlining them.
