# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

PoseStudio is an early-stage, open-source Qt6/C++ desktop application for 3D character creation, posing, and scene building (think a free alternative to DAZ Studio). The codebase is currently a small skeleton: a main window shell, an asset manager side panel backed by SQLite, a Preferences dialog, a menu bar, and the foundation of a Vulkan-backed 3D viewport (`src/viewport/` — device/swapchain/renderer scaffolding that draws a Blender-style infinite floor grid through a real camera and can import + render lit Wavefront `.OBJ` meshes; see the 3D Viewport section below). The rigging, animation, and final-frame rendering systems described in README.md are still aspirational/roadmap — they don't exist in code yet.

## Build

Requires Qt 6 (developed against 6.11.1, MSVC 2022 64-bit), CMake 3.21+, and the **LunarG Vulkan SDK** (https://vulkan.lunarg.com — supplies the Vulkan headers, loader, validation layers, and the `glslc` shader compiler). `find_package(Vulkan REQUIRED)` aborts configuration with a clear message if the SDK is absent, so the viewport can't be built without it.

```
cmake -B build -DCMAKE_PREFIX_PATH="C:/Qt/6.11.1/msvc2022_64"
cmake --build build
```

If `CMAKE_PREFIX_PATH` isn't passed, `CMakeLists.txt` falls back to the hardcoded path `C:/Qt/6.11.1/msvc2022_64` — never commit a change to that fallback to point at your personal Qt install.

GLM (math), AMD's Vulkan Memory Allocator (VMA), and tinyobjloader (the OBJ importer's parser) are pulled in header-only via CMake `FetchContent` at pinned tags (see the "3D VIEWPORT DEPENDENCIES" block) — a fresh clone needs only Qt + the Vulkan SDK, nothing vendored. tinyobjloader, like VMA, is a single-header lib whose implementation is compiled in exactly one TU (`scene/objloader.cpp` via `#define TINYOBJLOADER_IMPLEMENTATION`). They're *populated* (downloaded) but their own CMake isn't run; we consume their headers directly via `target_include_directories`. GLM's Vulkan-convention flags (`GLM_FORCE_RADIANS`, `GLM_FORCE_DEPTH_ZERO_TO_ONE`) are set once project-wide as `target_compile_definitions` — never per-file, or translation units disagree on matrix layout (a silent-corruption bug).

The build links `Qt6::Widgets`, `Qt6::Gui`, `Qt6::Sql`, and `Vulkan::Vulkan` (the loader; `QVulkanInstance`/`QWindow` live in Qt6::Gui). No Qt Network/3D/Quick modules. `CMAKE_AUTOMOC`/`AUTOUIC`/`AUTORCC` are all on, so new `Q_OBJECT` classes, `.ui` files, and resources just need to be added to the `add_executable` source list / `resources.qrc` — no manual moc/rcc invocation.

Two post-build mirror steps copy on-disk content next to the executable on every build (both true mirrors — `rm -rf` then `copy_directory`, not overlays, so renamed/removed source files don't linger): (1) `resources/Maquettes/` (the built-in asset library, real on-disk content — not embedded via `resources.qrc`, since the Asset Manager browses libraries through `QDir`, not Qt's resource filesystem); and (2) the compiled SPIR-V shaders. Shaders are GLSL under `src/viewport/shaders/`, compiled to `*.spv` by `glslc` via a per-shader `add_custom_command` (gated through the `PoseStudioShaders` custom target that `PoseStudio` depends on), then mirrored to `<applicationDirPath>/shaders` where `VulkanRenderer` loads them at runtime. Add a new shader by dropping the `.vert`/`.frag` in that folder and appending its name to the `SHADER_SOURCES` list in `CMakeLists.txt`.

**Cross-platform posture**: PoseStudio is meant to be fully cross-platform (Windows, macOS, Linux). The stack is portable by construction — Qt6, C++17, CMake, SQLite, Vulkan — and the codebase carries no `#ifdef` platform branches; platform differences are absorbed by Qt (window/surface creation via `QVulkanInstance`/`QWindow`) and CMake rather than hand-written per-OS code. Accommodations already in place: writable data lives in `QStandardPaths::AppDataLocation` (an installed app's own directory is read-only on macOS/Linux — see the Database section); the Windows icon resource (`resources/PoseStudio.rc`) is added to the target only `if(WIN32)`; and because macOS runs Vulkan through **MoltenVK** (a Vulkan-on-Metal translation layer), `VulkanContext` enables the spec-mandated `VK_KHR_portability_subset` device extension whenever a device advertises it (a no-op on Windows/Linux). **Caveat:** only the Windows/MSVC build has been exercised so far — the macOS/Linux paths are correct by construction but unbuilt/unrun, and the one known runtime unknown on macOS is whether Qt's `QVulkanInstance` auto-adds the *instance*-level portability-enumeration extension (verify on real Apple hardware before claiming macOS support).

There is no test suite, linter, or CI configuration in this repo yet.

## Architecture

**Source layout**: `src/` is organized into one folder per subsystem — `core/` (non-visual services: database, preferences manager), `shell/` (app chrome: menu bar, splash/About overlay, Fusion theming), `assets/` (Asset Manager), `preferences/` (the Preferences dialog + every panel), and `viewport/` (the 3D engine, itself split into `rendering/` + `scene/`). Only `main.cpp` and the cross-cutting `constants.h` sit at the `src/` root. Every subsystem folder is on the compiler include path (`target_include_directories` in `CMakeLists.txt`), so headers are included by bare name (`#include "database.h"`) regardless of which folder includes them — when you add a new `src/` subfolder, add it there too or its headers won't resolve.

**Startup sequence matters** (`src/main.cpp`): the SQLite database is opened and `PreferencesManager` is loaded *before* any widget is constructed, because widgets (e.g. `AssetManagerWidget`) read preferences/DB state during construction. If you add a new manager/cache that UI depends on, initialize it in this same early block, not lazily inside a widget.

**Database** (`src/core/database.{h,cpp}`): single named QSqlDatabase connection (`"db_conn"`) shared process-wide via `QSqlDatabase::database("db_conn")` rather than opened per-call. The SQLite file (`posestudio.db`) lives in the platform's per-user app-data dir (`QStandardPaths::AppDataLocation`, derived from the app name set in `main.cpp` *before* `initializeDatabase()` runs) — not next to the executable, whose directory is read-only for an installed app on macOS/Linux. Older builds stored it next to the `.exe`; `initializeDatabase()` performs a one-time *move*-migration of that legacy file into the new location on a Normal launch (copy-then-delete, so a later Factory Reset can't resurrect it), and FactoryReset also clears any lingering legacy file. Schema lives in `resources/database/initialize.sql`, embedded into the binary via `resources.qrc`, and is executed both on `DbInitMode::FactoryReset` (which also deletes the existing file first, and is reachable from the app via Preferences → Factory Reset) *and* on a genuine first launch (no `posestudio.db` file yet) — `initializeDatabase()` checks `QFile::exists(dbPath)` *before* FactoryReset can delete it, so a fresh install always gets the schema even via `DbInitMode::Normal`. There is still no formal migration system, but additive schema changes for existing databases (new columns/tables) are applied unconditionally at the end of `initializeDatabase()`, each guarded to be a no-op once already present (e.g. a `PRAGMA table_info` check before `ALTER TABLE ADD COLUMN`, or `CREATE TABLE/INDEX IF NOT EXISTS`) — add new ones here rather than only in `initialize.sql`, or pre-existing installs will silently fail at the first query touching the new column/table.

`AssetLibraries` has an `AssetLibraryIsBuiltIn` flag for the one library that ships with the app ("Maquettes", at `<applicationDirPath>/Maquettes`): `initializeDatabase()` re-syncs that row's path to the current install location on every launch (in case the app moved) and `mkpath`s the folder as a safety net. Built-in libraries always sort first (`ORDER BY AssetLibraryIsBuiltIn DESC`) and are excluded from the user-manageable list in Preferences → Assets, but still appear/behave like any other library in the Asset Manager tree.

`Favorites` (`FavoriteID`, `FavoritePath` unique, `FavoriteSortOrder`) is a flat list of favorited asset file paths — no folders, no nesting. `FavoriteSortOrder` holds the user's manual drag order from the Favorites pane (see Asset Manager section below); new favorites append after the current max.

**Preferences manager** (`src/core/preferencesmanager.{h,cpp}`): a process-wide singleton (`PreferencesManager::instance()`) that loads the entire `Preferences` SQLite table into a `QHash` once at startup. Reads never touch the DB; writes update the cache and persist immediately. It is explicitly not thread-safe — only call it from the Qt GUI thread.

**Asset manager** (`src/assets/assetmanagerwidget.{h,cpp}`, the largest file in the project): the side-panel widget that browses physical asset library folders (paths stored in `AssetLibraries`) alongside virtual, user-defined "Collections" (`AssetCollections`/`AssetCollectionItems`/`AssetCollectionFolders` tables) and a flat "Favorites" list (`Favorites` table). Collections can nest arbitrarily deep via `AssetCollections.AssetCollectionParentID` (`0` = top-level) — any collection, at any depth, can hold its own flat asset items and folder shortcuts alongside further sub-collections, mirroring filesystem folder semantics. Favorites, by contrast, is intentionally flat — no folders, no sub-favorites, just asset items. `contextForTreeItem()` walks a tree item's ancestor chain to classify it as `Library`/`Collection`/`SearchResults`/`Favorites` regardless of nesting depth; prefer it over checking a node's immediate parent when context-menu or grid logic needs to know which "world" an item belongs to. Tree root order is Search Results, Favorites, Collections, then physical libraries (built-in libraries first). Key pieces:
- `AssetFolderProxyModel` wraps the filesystem `QStandardItemModel`-backed tree to mark folders as direct/indirect "hits" (i.e. contain or descend into a folder containing matching assets) for search/filtering, with results debounced via a pending-index queue rather than recomputed synchronously.
- `AssetTreeDelegate` / `AssetGridDelegate` fully override Qt's default painting for the directory tree and the thumbnail grid respectively — don't expect standard `QStyledItemDelegate` behavior; geometry constants they rely on (icon size, margins, gaps) are centralized in `src/constants.h` so the two delegates' size hints can't drift apart.
- Breadcrumb rendering is cached per `displayFolder()` call (`m_breadcrumb*` members) and only re-laid-out, not recomputed, on label resize. Virtual sources (Collections, Favorites) instead show a plain title (`m_currentTitleText`); `displayFolder()` and `refreshTitleLabel()` both branch on the same "is this a virtual path" check (`COLLECTION_` prefix or `FAVORITES_ROOT`) — keep them in sync if a third virtual source is ever added.
- Physical folders and DB-backed virtual Collections/Favorites are all rendered into the same `QTreeView`/`dirModel`, distinguished by `searchResultsRootItem`/`favoritesRootItem`/`collectionsRootItem` root nodes — when adding tree behavior, check which subtree (filesystem vs. virtual) an index belongs to before assuming filesystem semantics (e.g. rename-on-disk) apply.
- `buildAssetHits()` resolves a flat list of asset paths into thumbnailed `AssetHit`s (shared by `parseCollectionAssets`/`parseFavorites`); it groups by folder internally for efficient thumbnail scanning but always re-emits in the caller's *input* order — both Favorites and Collections rely on this to preserve the user's manual drag order, so don't reintroduce a hash-ordered return here.
- Both Favorites and Collections are sortable grids backed by their own per-row order column (`Favorites.FavoriteSortOrder`, `AssetCollectionItems.AssetCollectionItemSortOrder`); `AssetManagerWidget::isSortableView()` (true when `m_currentFolderPath` is `FAVORITES_ROOT` or a `COLLECTION_*` id) is the single gate for both the drag-reorder handling and the "Sortable" info-bar label — check it rather than re-deriving the same `FAVORITES_ROOT`/`COLLECTION_` checks elsewhere. The drag-to-reorder itself (`beginGridDrag`/`endGridDrag`/`gridInsertIndex`/`updateGridDropIndicator`/`persistGridOrder` + the `MouseButtonPress/Move/Release` block in the grid viewport's `eventFilter`) is a **hand-rolled** drag, not Qt's `QListWidget` `InternalMove`: in `IconMode`, Qt's built-in internal-move free-positions the icon instead of reordering rows, and its `QDrag` runs through the OS drag loop. The manual version tracks raw mouse events, floats a translucent ghost `QLabel` plus a between-items drop-line indicator, leaves the grid untouched until release (moving items mid-drag would reflow the layout under the cursor), and auto-scrolls via a `QTimer` when the cursor sits in the top/bottom edge zone. `persistGridOrder()` branches on which view is current to write back to the right table/column. `AssetGridDelegate::paint` checks a `gridDragging` dynamic property on the grid to suppress the hover highlight while this is active.

**Styling**: dark theme enforced via `app.setStyle(new AppProxyStyle("Fusion"))` in `main.cpp`, never the native OS style. `AppProxyStyle` (`src/shell/appproxystyle.h`) must be constructed with `"Fusion"` explicitly — a bare `AppProxyStyle()` wraps the platform *default* style (e.g. `QWindows11Style` on Windows) instead, silently defeating the whole point of forcing Fusion. All app-level theming (style + global tooltip animation toggles) lives in this one spot in `main.cpp`'s "Branding & theming" block, not scattered into widget constructors. Stylesheets are modular (`global.qss`, `_assetmanager.qss`, `_menumanager.qss`, `_preferences.qss`) and concatenated in `loadStylesheets()` (`src/main.cpp`) in a fixed cascade order — later files override earlier ones, so module-specific overrides belong in the `_*.qss` files, not `global.qss`. Some custom widgets (e.g. `AssetTreeView`) expose `Q_PROPERTY` values (like `separatorColor`) specifically so QSS can drive C++ painting logic. `QDialog`/`QMessageBox` are dark-themed globally too — but note Qt's QSS engine doesn't apply selectors (e.g. `a { color: ... }`) to rich-text `<a>` links inside a `QLabel`; those need the color set inline in the HTML string itself (see the "Add Asset Folder" hint label in `assetmanagerwidget.cpp` for the pattern). Also note: avoid declaring *any* `border` (even `border: none;`) on a `QListView`/`QListWidget` `::item` rule unless you actually want one — doing so flips Qt to full CSS box-model rendering of that item, which can visibly inflate the selection/hover rect beyond the item's actual layout cell (this is why `#AssetManagerTree::item` and `#AssetLibraryList::item` deliberately have no `border` line). `outline: none;` is fine on a container (`#AssetManagerTree`, `#AssetLibraryList`) to kill the dotted focus ring, but don't assume it's a no-op everywhere in the cascade.

**Destructive operations** (e.g. Preferences → Factory Reset, in `src/preferences/factoryresetpreferencespanel.cpp`): rather than trying to reconcile every widget's in-memory state (the asset tree, `PreferencesManager`'s cache, etc.) after wiping the database, the app just relaunches itself via `QProcess::startDetached(QApplication::applicationFilePath(), ...)` followed by `QApplication::quit()`. Follow this pattern for any future action that invalidates broad in-memory state rather than writing ad-hoc refresh logic for every affected widget. The reset itself is gated inline on the page (type "RESET" into a text field to enable the button) rather than behind a separate confirmation popup.

**Preferences dialog** (`src/preferences/preferencesdialog.{h,cpp}`, `src/preferences/preferencespanel.{h,cpp}`, `src/preferences/*preferencespanel.{h,cpp}`): a modal `QDialog` with a vertical tab list (left) driving a `QStackedWidget` (right). Each tab is a `PreferencesPanel` subclass living in its own file (`AssetsPreferencesPanel`, `FactoryResetPreferencesPanel`, etc.) so areas can be built out independently — adding a tab means adding a new subclass file and one `addPanel()` call in `PreferencesDialog`'s constructor, not touching the others. `PreferencesPanel` provides shared chrome (heading + top-aligned content layout) plus `addDescription()`/`addPlaceholder()` helpers for standardized body-copy styling — use these instead of hand-rolling a `QLabel` with inline font-size, or the panel won't match the others. `MenuManager::openPreferencesDialog(initialTab)` is the one entry point that opens the dialog (Edit → Preferences, and the Asset Manager's "Manage Asset Folders" context actions both route through it) and wires the `AssetsPreferencesPanel`'s `librariesChanged`/`navigateToLibraryRequested` signals back to the live `AssetManagerWidget` — `MenuManager::setAssetManagerWidget()` must be called (from `main.cpp`, after the Asset Manager tab exists) before that wiring can attach, so don't move `setupMenus()` to run after it without checking this dependency.

**3D viewport** (`src/viewport/`): a Vulkan renderer, hosted in a custom `QWindow` (not Qt's `QVulkanWindow` — we deliberately own the device/swapchain/render passes for the flexibility advanced rendering will need). The subsystem is split into a thin Qt-facing layer and a **Qt-free rendering core**, and that boundary is load-bearing — keep it:
- `ViewportWidget` (`viewportwidget.{h,cpp}`) is the *only* class the rest of the app sees. It's a plain `QWidget` that owns the process's `QVulkanInstance`, embeds the `VulkanWindow` via `QWidget::createWindowContainer()`, and degrades to an inline `QLabel` (rather than crashing the app) if `QVulkanInstance::create()` fails. `main.cpp` just news one up where the old viewport placeholder was. It also owns the teardown *ordering*: its destructor explicitly `delete`s the `createWindowContainer` wrapper (hence the `VulkanWindow` and all its Vulkan objects) **while the `QVulkanInstance` member is still alive**. Without that, the base `QWidget` destructor would free the window *after* the `QVulkanInstance` member is gone, and the renderer would call `vkDestroyDevice` on a device whose instance was already destroyed — a crash on close. Don't drop that explicit delete. **Overlay gotcha:** because the viewport is a *native* child window (`createWindowContainer`), it composites on top of ordinary sibling/overlay `QWidget`s — so anything that must paint over the viewport (the boot/About `SplashOverlay`, any future HUD) must be a **top-level window** (constructed with the `Qt::Window` type flag, then positioned over the parent in global coords), not a child overlay. Window *hints* alone (`FramelessWindowHint`, etc.) don't promote a parented widget to a window — `isWindow()` stays false and it remains hidden behind the viewport.
- `VulkanWindow` (`vulkanwindow.{h,cpp}`, a `QWindow` with `surfaceType == VulkanSurface`) is the Qt↔Vulkan bridge and the *only other* file in the subtree allowed to include Qt. It lazily builds the `VulkanContext` + `VulkanRenderer` on first `exposeEvent` (the platform surface must exist first), pumps one frame per `QEvent::UpdateRequest` (re-arming `requestUpdate()` each frame for continuous redraw), and routes mouse/wheel input to the camera. **Camera drags are gated on `m_activeDragButtons`** — buttons whose press was actually seen *in this window* — not on `QMouseEvent::buttons()`: a modal dialog (e.g. the Import file picker) closing over the native child window can otherwise leak a button-held move to us and snap the camera, so a move only orbits/pans if its button's drag began here. It releases its Vulkan objects both on `QEvent::PlatformSurface`/`SurfaceAboutToBeDestroyed` (so they're gone before Qt frees the surface they reference) and in its destructor as a fallback — `releaseVulkan()` is idempotent. **Header gotcha:** `QPlatformSurfaceEvent` is pulled from `<QtGui/qevent.h>`, *not* `<qpa/qplatformsurfaceevent.h>` — the binary Qt install ships no qpa headers, so the seemingly-canonical qpa include fails to compile here.
- Everything under `rendering/` and `scene/` is **pure Vulkan + std + GLM, no Qt includes** (it takes `std::string` paths and `std::vector<char>` SPIR-V blobs, not `QString`/`QByteArray`). This keeps the engine core reusable and testable. Don't reach for a `Q*` type here just because it's convenient — push that concern up into `VulkanWindow` (e.g. `importObj` takes a `QString` at the `VulkanWindow`/`ViewportWidget` layer but hands a `std::string` to the renderer).

Ownership/lifetime is layered by how often things change: `VulkanContext` (instance handle, physical+logical device, queues, VMA allocator) is created once and lives until the surface dies. `VulkanSwapchain` (swapchain images, the shared depth buffer, render pass, framebuffers) is the resolution-dependent layer — `recreate()` rebuilds it on resize/out-of-date but deliberately *keeps the render pass* (it depends only on formats, which don't change). `VulkanRenderer` owns the per-frame state (command buffers, sync) and the swapchain plus the renderables — the `Scene` (imported meshes) and the `Grid` — and drives `acquire → record → submit → present`. Resize is coalesced: `notifyResize()` just sets a pending extent + flag, and the actual rebuild happens at the next `drawFrame()` (on `VK_ERROR_OUT_OF_DATE_KHR`/suboptimal/the flag) after a `vkDeviceWaitIdle`. Sync follows the standard pattern with one subtlety — `imageAvailable` semaphores and `inFlight` fences are per *frame-in-flight* (`kMaxFramesInFlight`, in `vulkancommon.h`), but `renderFinished` semaphores are per *swapchain image* (a submit-signalled semaphore mustn't be reused until that image's present completes), so they're resized whenever the swapchain image count changes.

`VK_CHECK` (`vulkancommon.h`) throws `VulkanError` on any non-success `VkResult`; `VulkanWindow::initializeVulkan()` is the catch boundary. VMA is header-only and must be compiled in exactly one TU — `vma_impl.cpp` is that file (`#define VMA_IMPLEMENTATION`), bound to the statically-linked loader (`VMA_STATIC_VULKAN_FUNCTIONS 1`), which is why `VulkanContext` can call `vmaCreateAllocator` without supplying a function-pointer table.

`VulkanPipeline` wraps one `VkPipeline` + layout. The *shared* fixed-function state (dynamic viewport/scissor — so pipelines survive resizes untouched — triangle-list topology) lives in the class; the knobs that differ between pipelines are passed in a small `PipelineConfig` (push-constant size/stages, alpha-blend on/off, depth test/write on/off, cull mode, plus optional vertex-input bindings/attributes and descriptor-set layouts). Add a field there only when a real use case needs it — don't expose all of Vulkan's pipeline state speculatively. Two pipelines exist today, drawn in this order inside the render pass:
- **Mesh** (`scene/scene.{h,cpp}` + `mesh.vert/frag`): the lit pipeline for imported geometry. Interleaved `Vertex` (pos/normal/uv) vertex buffers, a per-frame camera/lighting UBO in descriptor **set 0** (binding 0, vert+frag), a per-material diffuse texture in descriptor **set 1** (combined image sampler, fragment), and an 80-byte push constant (`MeshPushConstants` = `mat4 model` + `vec4 baseColor`) per draw. Fragment albedo = `baseColor * texture(uv)`. Opaque, depth test+write, **no back-face culling** — imported OBJ winding is inconsistent, so the fragment shader instead flips the normal per `gl_FrontFacing` for correct two-sided lighting. Drawn first (it's the opaque layer).
- **Floor grid** (`scene/grid.{h,cpp}` + `grid.vert/frag`): a Blender-style *infinite* ground grid. It is NOT line geometry — it draws one full-screen quad and the fragment shader reconstructs the world point on the `y=0` plane per pixel (ray/plane intersect of the unprojected near/far points), then draws anti-aliased lines via `fwidth` derivatives and fades them out with distance. It writes correct `gl_FragDepth` (so scene geometry occludes it) but is alpha-blended with depth-*write off* (a transparent overlay mustn't occlude things behind the mostly-invisible plane), which is why it's drawn last (after the opaque mesh layer). Its push constant is a single `viewProj` matrix, visible to both stages (vertex inverts it to unproject; fragment uses it for depth). All look tuning (line colour, cell size, major-line interval, fade distance) lives as named constants at the top of `grid.frag`; there are no colored axis lines yet.

**Mesh rendering & model import**: `Scene` (`scene/scene.{h,cpp}`) is the mesh analogue of `Grid` — it owns the mesh pipeline, the per-frame-in-flight camera UBOs + descriptor sets (set 0), the shared set-1 layout + a 1×1 white fallback texture, and the list of imported `Model`s, exposing `addModel()` and a `record(cmd, camera, frameIndex)` the renderer calls before the grid. A `Model` (`scene/mesh.{h,cpp}`) is one imported file: a set of `Mesh`es (one per material group) under a shared transform, and it owns the `VkDescriptorPool` its meshes' set-1 descriptors come from (sized to its mesh count). Each `Mesh` owns device-local vertex+index buffers, a base color, an optional `VulkanTexture`, and a set-1 descriptor pointing at that texture (or the scene's white fallback when untextured — so one pipeline/shader serves both: `albedo = baseColor * white = baseColor`). GPU buffers go through `VulkanBuffer` (`rendering/vulkanbuffer.{h,cpp}`); textures through `VulkanTexture` (`rendering/vulkanimage.{h,cpp}`, an `_SRGB` sampled image with a full mip chain). Both upload via `submitImmediate` (`rendering/vulkancommands.{h,cpp}`), a one-shot graphics-queue submit + fence wait used for all load-time transfers/transitions.

Image decoding is deliberately **not** in the Qt-free core: `loadObj()` (`scene/objloader.{h,cpp}`, the single tinyobjloader TU) parses an OBJ+MTL into CPU `ModelData` (de-indexed unique `Vertex`es, smooth normals when the file lacks them, grouped by material, with `Kd` as base color and the *resolved path* of `map_Kd`) — pure data, no Vulkan, no codec. The **Qt layer decodes the texture files**: `VulkanWindow::loadAndAddModel` runs `loadObj`, then `QImage`-decodes each `diffuseTexturePath` into tightly-packed RGBA8 (`MeshData::diffusePixels`), then calls `VulkanRenderer::addModel(ModelData)`. This keeps the renderer free of any image-codec dependency while supporting every format Qt's image plugins do. The full chain is `MenuManager` (File → Import → `.OBJ`, a `QFileDialog`) → `ViewportWidget::importObj` → `VulkanWindow::importObj` (queues the path if the renderer isn't built yet — it's created lazily on first expose) → `loadAndAddModel` → `VulkanRenderer::addModel` → `Scene::addModel`. `main.cpp` also feeds any `.obj` on the command line through `ViewportWidget::importObj` (drag-onto-exe / "open with"), which is why the queue-until-ready step matters. Imports run on the GUI thread between frames; the staging uploads block briefly on their own submits but touch no in-flight state, so the next recorded frame just picks the new model up. **Not yet:** non-diffuse PBR maps (normal/metal/rough — OBJ doesn't natively carry them), and the other 8 Import formats are still disabled placeholders.

The `Camera` (`scene/camera.{h,cpp}`) is a pure-GLM orbit camera; its projection negates `[1][1]` for Vulkan's flipped clip-space Y on top of the project-wide `GLM_FORCE_DEPTH_ZERO_TO_ONE`. Natural next increments: more importers (glTF/USD), and richer materials (PBR map slots beyond the diffuse texture — normal/metal/rough — which need a richer source format than OBJ) — each slots into the existing `Scene`/`Mesh`/loader/`VulkanTexture` modules without disturbing the others.

**Constants** (`src/constants.h`): shared UI dimensions, colors, and timing values live here rather than being duplicated as magic numbers across widgets/delegates — add new shared layout/color/timing constants here instead of inlining them. (Viewport-internal tuning that no other subsystem needs — input sensitivity, frames-in-flight — currently lives next to its use in the viewport files; promote a value here only once something outside `src/viewport/` needs it.)
